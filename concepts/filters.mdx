description: "Strongram's Filter system lets you declare exactly which updates a handler accepts using 40+ pre-built predicates combined with and(), or(), and negate()."
---

A `Filter` is a `Predicate<RequestContext>` with one addition: its `and()`, `or()`, and `negate()` methods return `Filter` instead of `Predicate`, so you can chain calls fluently and the result is always a `Filter` ready to be returned from `getFilter()`.

```java
public interface Filter extends Predicate<RequestContext> {
    Filter and(Predicate<? super RequestContext> other);
    Filter or(Predicate<? super RequestContext> other);
    Filter negate();
}
```

## Using a filter in a handler

Return a filter from `getFilter()` in any `FilteredUpstreamHandler`. The framework evaluates it before calling `processFiltered`; updates that do not match are automatically forwarded to the next handler.

```java
public class PrivateTextHandler extends FilteredUpstreamHandler {
    @Override
    protected Filter getFilter() {
        return Filters.isPrivateChat().and(Filters.hasMessageText());
    }

    @Override
    protected void processFiltered(RequestContext ctx) {
        ctx.getResponder().send("Got your message.");
    }
}
```

## The `Filters` factory

`Filters` is a static utility class with pre-built filter constructors organized by category.

### Wildcards

| Method | Matches |
|---|---|
| `acceptAll()` | Every update |
| `rejectAll()` | No update |

### Text matching

All text-matching filters use the text extracted by `As.text()`, which covers message text and callback data.

| Method | Matches |
|---|---|
| `textStartWith(String s)` | Text starts with `s` |
| `textEndsWith(String s)` | Text ends with `s` |
| `textContains(String s)` | Text contains `s` (case-sensitive) |
| `textContainsIgnoreCase(String s)` | Text contains `s` (case-insensitive) |
| `textEquals(String s)` | Text equals `s` exactly |
| `textEqualsIgnoreCase(String s)` | Text equals `s` (case-insensitive) |

### Identity

| Method | Matches |
|---|---|
| `userIdIs(Long id)` | `uuid.user().getId()` equals `id` |
| `chatIdIs(Long id)` | `uuid.chat().getId()` equals `id` |
| `uuidEquals(TelegramUUID)` | UUID equals the given value |
| `hasUser()` | `uuid.user()` is not null |
| `hasChat()` | `uuid.chat()` is not null |

### Combinators

| Method | Description |
|---|---|
| `iterateOr(Function<T, Filter>, T first, T... rest)` | Applies the factory to each value and combines results with `or` |
| `iterateAnd(Function<T, Filter>, T first, T... rest)` | Applies the factory to each value and combines results with `and` |

## Composing filters

Use `and()`, `or()`, and `negate()` to build composite conditions:

```java
// Private chat that has text
Filter filter = Filters.isPrivateChat().and(Filters.hasMessageText());

// Not a command
Filter filter = Filters.isCommand().negate();

// Photo or document
Filter filter = Filters.hasPhoto().or(Filters.hasDocument());
```

## Multi-value matching with `iterateOr` and `iterateAnd`

`iterateOr` is useful when a handler should accept any one of several specific values:

```java
// Accept "yes", "no", or "maybe" (case-insensitive)
Filter filter = Filters.hasMessageText().and(
    Filters.iterateOr(
        Filters::textEqualsIgnoreCase,
        "yes", "no", "maybe"
    )
);
```

`iterateAnd` applies the factory to each value and requires all results to be true — use it when a single piece of text must satisfy several independent predicates simultaneously.

<Note>
`iterateOr` and `iterateAnd` require at least one value argument in addition to the factory. Passing zero values will not compile because the first value parameter is not variadic.
</Note>
